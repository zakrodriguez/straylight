## Answer key

<details>
<summary>Click to expand</summary>

**Q1.** **b)** The `ICertRequest2` interface is reachable with the
caller's credentials. It proves DCOM/RPC reach plus the DCOM
activation ACL â€” and nothing about template permissions or
issuance success.

**Q2.** **b)** Single-digit milliseconds. >200 ms suggests DNS or
DCOM endpoint cache issues; first call may be slower as the cache
warms.

**Q3.** `dNSHostName` (must resolve to the CA server) and
`cACertificateDN` (must match the subject of the CA's signing
certificate). Wrong `dNSHostName` and clients can't find the CA;
wrong `cACertificateDN` and the discovered cert doesn't match the
one the CA actually presents.

**Q4.** **b)** Production enrollment requests come from non-admin
contexts (autoenrollment, service accounts). The low-priv ping
exercises the enrollment-interface DCOM ACL that those requests
will hit; admin contexts bypass parts of that ACL.

**Q5.** Template = `CAExchange`. EKU = `Private Key Archival`,
OID `1.3.6.1.4.1.311.21.5`. (Validity is in days, not years â€” it's
deliberately short-lived.)

**Q6.** **b)** Auto-generated by the CA, self-issued from the
`CAExchange` template, callable by any authenticated user. No
permissions to set up, no INF to write, no template publishing
required.

**Q7.** **c)** Revocation server offline â€” `certutil` could not
fetch a current CRL (or any CRL) within the timeout window. Often
the downstream effect of `0x80072EFD` (TCP failure to the CDP
host) or `0x80072EE7` (DNS failure).

**Q8.**
- `0x80072EE7` â†’ DNS lookup failed
- `0x80072EFD` â†’ TCP connect failed
- `0x80092013` â†’ Revocation server offline
- `0x800B010A` â†’ Chain not trusted (root not in store)

**Q9.** **b)** A functional test has to reflect current network
reality. A cached CRL or chain artifact from an earlier run would
mask current breakage; `-urlfetch` deliberately re-resolves every
URL.

**Q10.** Most likely `0x80072EE7` (DNS) or `0x80072EFD` (TCP)
against the **HTTP** CDP/AIA URLs. Linux validators usually only
consult HTTP CDPs; Windows clients also have an LDAP CDP entry
that goes to `dc1` and resolves successfully even if HTTP is
broken. So the Windows chain still builds but the Linux chain
doesn't â€” the textbook "HTTP CDP broken, LDAP CDP still working"
asymmetric failure.
